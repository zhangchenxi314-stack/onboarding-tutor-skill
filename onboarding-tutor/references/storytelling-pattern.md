# Storytelling Pattern（故事化叙事模式）

## 核心结构：点 → 面 → 点

讲解必须形成一个闭环：从一个具体场景切入，展开到全局体系，再回归到具体场景解决实际问题。

```
点（切入）  →  面（展开）  →  点（回归）
具体场景       全局体系        解决原场景
```

## 为什么用这种结构

- **点切入**：降低认知门槛，新人对具体场景有共鸣
- **面展开**：建立全局观，理解"为什么这么设计"
- **点回归**：验证学习效果，能解决实际问题

## 模式 A：故障驱动（Debug-Driven）

**适用场景**：解释链路、排查流程、异常处理机制

**点（切入）**：
> "假设你收到一条告警：`订单创建超时，耗时 5200ms，阈值 3000ms`。你打开监控面板，发现库存服务的 P99 延迟从平时的 50ms 飙升到了 2000ms。你会怎么排查？"

**面（展开）**：
> "这个告警的背后，涉及订单服务的调用链路、库存服务的内部实现，以及整个系统的超时与降级机制。让我们从上游到下游完整梳理一遍：
> 1. 订单创建链路中，库存预占是同步调用——这是性能瓶颈的根源
> 2. 库存服务的内部实现：先查缓存、再查DB、加分布式锁——每一步都可能成为瓶颈
> 3. 系统如何感知和应对：超时配置、熔断器、降级策略..."

**点（回归）**：
> "回到那条告警。现在你应该能判断：库存服务延迟飙升 → 订单服务同步等待 → 累积超时。排查方向：
> 1. 检查库存服务是否在做大批量操作
> 2. 检查 Redis 连接池是否耗尽
> 3. 短期止血：调整订单服务的超时阈值或降级开关
> 4. 长期方案：将库存预占改为异步确认"

## 模式 B：演进驱动（Evolution-Driven）

**适用场景**：解释架构设计、技术选型、重构原因

**点（切入）**：
> "看这段代码：一个 800 行的 Service 类，里面混杂了参数校验、业务逻辑、DB 操作、缓存更新、消息发送。你接到需求要加一个新功能，你会直接把逻辑写进去吗？"

**面（展开）**：
> "这个 800 行的类，是典型的「大泥球」——业务演进中自然形成的。让我们看看一个健康的系统应该怎么分层：
> 1. Controller 层：参数校验 + 路由，不超过 50 行
> 2. Service 层：业务编排，不直接操作 DB
> 3. Repository 层：数据访问，单一职责
> 4. 为什么分层：可测试性、可替换性、团队并行开发..."

**点（回归）**：
> "回到那个 800 行的类。现在你应该能拆解它：
> 1. 抽离校验逻辑 → Request Validator
> 2. 抽离 DB 操作 → Repository
> 3. 缓存和消息发送 → 独立的 Domain Event + Handler
> 4. 新的需求如果只是新增一个校验规则，改 Validator 就行，不需要碰 800 行的类"

## 模式 C：对比驱动（Comparison-Driven）

**适用场景**：解释技术选型、多个方案的区别

**点（切入）**：
> "Leader 让你评估两个方案：方案 A 用 Kafka 做订单状态流转，方案 B 用数据库状态机 + 定时任务。你只有半天时间出结论，该关注哪些维度？"

**面（展开）**：
> "这不是一个简单的 A vs B 问题。方案选择的背后是对系统需求的深入理解：
> 1. [对比表：Kafka vs DB 状态机]
> 2. 我们的业务特点：订单量级、实时性要求、团队技术储备
> 3. 现有的基础设施：是否已有 Kafka 集群？运维成本如何？"

**点（回归）**：
> "半天时间足够出结论了：
> 1. 如果日订单 < 10w 且团队对 Kafka 不熟悉 → 方案 B，简单可控
> 2. 如果日订单 > 100w 或需要多消费者 → 方案 A，扩展性好
> 3. 向 Leader 汇报时，带上对比表和你的推荐理由，而不是只说选 A 或 B"

## 叙事技巧

1. **切入场景必须具体**：有人物（你/小明）、有事件（告警/需求/Bug）、有紧迫感（半天/马上）
2. **展开时用"为什么"串联**：不仅讲"是什么"，更要讲"为什么这么设计"
3. **回归时明确可操作**：给出具���的排查步骤、代码位置、优化方向
4. **时长控制**：点（15%）→ 面（60%）→ 点（25%），不要头重脚轻

## 注意事项

1. 不要编造不存在的场景——从用户提供的资料中找真实案例
2. 场景不要过于复杂——新人无法代入
3. 面≠漫谈——始终围绕切入场景的"为什么"展开
4. 回归≠重复——必须给新信息（排查方向/解决方案）
