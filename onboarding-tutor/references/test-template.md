# Test Template（选择题测试模板）

## 出题规则

### 题量
- 每个主题 3-5 道选择题
- 题目按难度递进：概念理解（简单）→ 流程判断（中等）→ 场景分析（较难）

### 题型

**单选题**：4 个选项中 1 个正确答案
**多选题**：4-5 个选项中 2-3 个正确答案（需明确标注"多选"）

### 选项设计原则
1. 所有选项必须"看起来合理"——不能有显然错误的选项
2. 干扰项来自常见的理解偏差（将资料中的易混淆点故意做成错误选项）
3. 选项长度相近，避免正确答案明显长/短于其他选项
4. 多选题至少要有 1 个强干扰项

### 不得出的题型
- 判断题（太简单，50%猜对概率）
- 填空题（主观性太强）
- 开放题（评价标准不统一）

## 出题模板

### 单选题模板

```
**单选题**：关于订单服务超时重试的策略，以下哪个是正确的？

A. 超时后立即重试 3 次，每次间隔固定 1 秒
B. 超时后使用指数退避重试，最多 3 次，总时间不超过 10 秒
C. 超时后不重试，直接返回失败给上游
D. 超时后无限重试直到成功

正确答案：B
```

### 多选题模板

```
**多选题**：以下哪些是库存预占采用"先预占后确认"模式的原因？（选择 3 项）

A. 避免超卖——先占住库存，支付完成再确认扣除
B. 提升用户体验——下单时就能锁定库存，用户有安全感
C. 减少数据库写操作——预占不写 DB
D. 支持订单超时释放——超时未支付自动回滚预占
E. 简化代码——比实时扣减实现简单

正确答案：A、B、D
```

### 场景题模板

```
**单选题**：某次大促期间，监控显示订单服务 P99 延迟飙升至 5 秒。以下排查思路中，最先应该执行的是？

A. 直接重启订单服务的所有实例
B. 检查慢查询日志，看是否是 DB 查询变慢
C. 检查库存服务的调用耗时和熔断器状态
D. 增加订单服务的实例数量

正确答案：C
解析：订单服务的主要下游依赖是库存服务（同步调用）。根据"先下游后上游"的排查原则，应先检查下游依赖的耗时和熔断状态再排查自身问题。A 是盲操，B 是次优排查方向，D 是扩容而非排查。
```

## 提交与评分格式

用户一次性提交所有答案，格式：

```
1A 2BD 3C 4A 5D
```

Agent 评分回复模板：

```
测试结果：3/5 正确（60%）

✓ 第1题 正确 — 订单状态机理解到位
✗ 第2题 错误 — 你的答案：ACD，正确答案：ABD
  解析：C "减少数据库写操作"不正确。预占模式在预占和确认两个阶段都需要写 DB，
  实际是增加了写操作。预占模式的核心优势是防止超卖和支持超时释放。

✓ 第3题 正确
✗ 第4题 错误 — 你的答案：A，正确答案：C
  解析：[具体解释为什么选C]

✓ 第5题 正确

薄弱点：对异步消息机制的理解需要加强（第2题），排查思路需要从"先自身后下游"纠正为
"先下游后自身"（第4题）。
```

## 评分后行动

| 正确率 | 处理 |
|--------|------|
| 100% | 告知正确率，指出仍需注意的易混淆点，进入下一主题 |
| 60%-80% | 指出错误原因，建议重新学习薄弱点的相关文档 |
| <60% | 定位薄弱根源，建议重新学习该主题的核心部分，标记为"待巩固" |

**注意**：不论正确率多高，都不要说"很棒""很不错"。只说事实。
