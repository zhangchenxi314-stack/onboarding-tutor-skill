# onboarding-tutor Skill 开发报告

> 版本: 1.0.0 | 日期: 2026-06-16 | 作者: Hermes Agent

---

## 1. 项目背景

### 1.1 需求来源

团队新人入职后，面对复杂的业务系统和文档，存在以下痛点：

- 文档分散（README、Wiki、设计文档、代码注释），缺乏系统性的学习路径
- 自学效率低：看了文档但抓不住重点，不理解模块间的关联
- 缺乏互动：无法验证自己是否真的理解了
- 提问门槛高：不知道问谁、问什么、怎么问

### 1.2 需求定义

开发一个 Hermes Skill，使 AI Agent 具备业务上手导师的能力，满足以下 8 项需求：

| # | 需求 | 核心诉求 |
|---|------|----------|
| 1 | 快速上手业务框架流程 | 降低新人学习成本 |
| 2 | 用户丢资料，AI 引导提问 | 互动式学习，而非被动阅读 |
| 3 | 时序图 + 表格增强理解 | 可视化技术路线 |
| 4 | 递进分点逻辑 | 有层次的知识结构 |
| 5 | 故事化叙事（点→面→点） | 降低认知门槛，建立全局观 |
| 6 | 客观评价（不总是鼓励） | 真实反馈，帮助成长 |
| 7 | 追问有上限 | 避免死磕，保证学习效率 |
| 8 | 选择题客观测试 | 可验证的学习成果 |

## 2. 架构方案选型

### 2.1 Skill 承载形式

| 方案 | 描述 | 优点 | 缺点 | 结论 |
|------|------|------|------|------|
| **A: Skill 注入（采用）** | 将教学 persona + 规则直接写入 SKILL.md，加载后由 LLM 在 prompt 中遵循 | 零开发成本；LLM 原生遵循；跨模型通用；可随时修改实时生效 | 依赖 LLM 遵循指令的能力；长 prompt 占用上下文 | ✅ 采用 |
| B: MCP Server | 开发独立的 MCP 服务，由 Agent 通过工具调用访问教学逻辑 | 状态管理精确；测试流程可控；与 LLM 解耦 | 开发成本高（需 Python/Node 服务）；部署维护负担；每次教学需网络调用 | ❌ 过度工程 |
| C: Persona 文件 | 写入 `~/.hermes/personas/` | 简单 | 无法包含交互协议、规则、模板等复杂逻辑；persona 文件不支持 references/ 子目录 | ❌ 功能不足 |
| D: Cron Job | 定时触发教学流程 | 适合定期培训 | 无法进行互动式问答；教学节奏不可控 | ❌ 场景不匹配 |

**决策依据**：本 skill 的核心价值在于"指令遵循"——让 LLM 在每次回复中都按照教学协议行事。Skill 注入是 Hermes 最原生的方式，无需额外开发，且支持 references/ 子目录存放模板。

### 2.2 Skill 所属分类

| 方案 | 描述 | 结论 |
|------|------|------|
| **productivity（采用）** | 生产力工具类：文档、教学、培训 | ✅ 教学属于生产力场景 |
| software-development | 软件开发类 | ❌ 教学不限于软件开发 |
| autonomous-ai-agents | AI Agent 自身 | ❌ 非 Agent 自身功能 |
| note-taking | 笔记类 | ❌ 这是教学工具，非笔记 |

### 2.3 知识可视化方案

| 方案 | 描述 | 优点 | 缺点 | 结论 |
|------|------|------|------|------|
| **A: Mermaid 代码块（采用）** | LLM 输出 Mermaid 语法代码块 | 纯文本，零依赖；GitHub/Notion 原生渲染；LLM 能直接生成 | 终端不可见（需在浏览器/IDE中查看） | ✅ 采用 |
| B: ASCII 图 | 纯文本字符画 | 终端可见 | 复杂流程难以表达；LLM 生成质量不稳定 | ❌ 时序图效果差 |
| C: 图片生成（Image Gen） | 调用 DALL-E/Midjourney 生成流程图 | 视觉效果好 | 耗时（10-30s）；内容不可控；成本高 | ❌ 不可靠 |
| D: Excalidraw JSON | 生成手绘风格 JSON | 可交互编辑 | 用户需要额外打开 Excalidraw；非标准格式 | ❌ 学习成本高 |

**决策依据**：Mermaid 是业内标准，LLM 生成质量稳定，且在主流的 Markdown 查看器（GitHub、Notion、Obsidian、VS Code）中都能原生渲染。对于复杂流程，Mermaid 的 sequenceDiagram 语法表达力最强。

### 2.4 测试题型方案

| 方案 | 优点 | 缺点 | 结论 |
|------|------|------|------|
| **A: 选择题（采用）** | 客观、自动评分、降低用户门槛、答案明确 | LLM 可能出题质量不稳定（需要在 skill 中约束） | ✅ 采用 |
| B: 填空题 | 比选择题更有区分度 | 语义相似但表述不同的答案难以自动判定正误 | ❌ 评分不可靠 |
| C: 开放题 | 能考察深层理解 | 评分主观、需要 LLM 二次判断、用户的抗拒心理强 | ❌ 不适合新人 |
| D: 编程题 | 实战能力考察 | 需要运行环境、耗时长、偏离"理解业务"的目标 | ❌ 偏离目标 |

**决策依据**：选择题降低用户的心理门槛（不需要长篇打字），答案明确可自动评分，且通过精心设计的干扰项（常见理解偏差）能有效检测理解程度。

## 3. Skill 架构设计

### 3.1 文件结构

```
~/.hermes/skills/productivity/onboarding-tutor/
├── SKILL.md                          # 主文档：5 阶段交互协议 + 4 条讲解规则 + 评价/测试/追问协议
└── references/                       # 辅助模板
    ├── sequence-diagram-guide.md     # Mermaid 时序图规范（含各种场景的模板）
    ├── table-format-guide.md         # 表格排版规范（4 种表格类型模板）
    ├── storytelling-pattern.md       # 3 种故事化叙事模式（故障/演进/对比驱动）
    └── test-template.md              # 选择题出题与评分模板
```

### 3.2 设计原则

| 原则 | 说明 |
|------|------|
| **核心精简，模板外置** | SKILL.md 8-12k 字符，详细模板放 references/，减少 prompt 占用 |
| **规则可验证** | 每条规则都有「正确示范」和「错误示范」对比 |
| **协议覆盖全流程** | 5 个阶段从材料接收到总结反馈，无缝衔接 |
| **禁止优于建议** | 用「禁止用语」明确约束 LLM 的鼓励倾向，比「建议客观」更有效 |

### 3.3 交互协议设计

```
阶段一：材料接收与初判
  ├── 确认接收
  ├── 初判摘要（≤5 句话）
  ├── 知识摸底（2-3 题）
  └── 生成学习路径
        │
阶段二：要点提炼与引导提问
  ├── 提炼 3-5 个核心要点
  ├── 生成引导问题（递进设计）
  └── 等待用户回答
        │
阶段三：深度讲解（遵循 4 条规则）
  ├── 规则 1：时序图优先
  ├── 规则 2：表格结构化
  ├── 规则 3：递进分点（概念→机制→关联→实践）
  └── 规则 4：故事化叙事（点→面→点）
        │
阶段四：客观测试
  ├── 出题（3-5 道选择题）
  ├── 评分（正误 + 解析）
  └── 薄弱点分析
        │
阶段五：总结与反馈
  ├── 知识点回顾
  ├── 薄弱点建议
  └── 下一主题预告
```

### 3.4 追问上限机制

```
第 1 轮追问  →  详细讲解 + 追问确认
第 2 轮追问  →  换角度讲解 + 追问确认
第 3 轮追问  →  最简化答案 → 标记"待巩固" → 强制推进到下一主题
```

同一主题超过 3 轮后不再接受追问，避免陷入死循环。用户主动提出的新角度问题不计入上限。

## 4. 开发过程

### 4.1 开发流程

```
需求分析 → 方案选型 → 编写 plan → 创建 SKILL.md → 编写 references/ → 验证
  (30min)    (20min)    (15min)      (40min)          (30min)        (10min)
```

### 4.2 关键决策记录

| 决策点 | 选项 | 选择 | 理由 |
|--------|------|------|------|
| Skill 主文档长度 | 5k / 10k / 20k | ~10k | 规则完备但不冗长，留给 LLM 足够思考空间 |
| 评价风格 | 温和 / 严格 / 中性 | 严格 | 用户明确要求"不能总是鼓励"，且严格评价更有助于新人真正理解薄弱点 |
| 故事模式数量 | 1 种 / 3 种 / 5 种 | 3 种 | 覆盖主要场景（故障/演进/对比），太多反而增加 LLM 选择困难 |
| 出题数量 | 3 题 / 5 题 / 10 题 | 3-5 题 | 太少覆盖不全，太多疲劳。3-5 题是黄金区间 |

## 5. 风险与局限

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| LLM 不严格遵循协议 | 教学行为偏离预期 | SKILL.md 中大量使用正确/错误示范对比，提高遵循率 |
| Mermaid 在终端不可见 | 用户看不到时序图 | 在 SKILL.md 中注明用户可在支持 Mermaid 的查看器中打开 |
| 选择题质量不稳定 | 测试无效 | references/test-template.md 提供详细的出题规范和正反示例 |
| 长时间对话超出上下文 | 丢失前面的协议指令 | 依赖 Hermes 的 context compression 机制，SKILL.md 本身较小 |
| 教学节奏过慢 | 用户不耐烦 | 追问上限机制确保不卡在单个点，用户可以随时要求跳过 |

## 6. 后续迭代方向

1. **多主题管理**：支持用户同时学习多个模块，记录学习进度
2. **自适应难度**：根据用户测试结果自动调整讲解深度
3. **学习报告导出**：将测试结果和薄弱点导出为文档
4. **代码实战集成**：对于 coding 类业务，增加编程练习环节
5. **语音交互**：结合 Hermes 的 STT/TTS 能力，支持语音教学
